# 🤖 GridBot 2026 - Архитектура и Контекст для ИИ

Этот файл содержит полное описание текущей архитектуры, логики и особенностей проекта **GridBot**. Его цель — быстро погрузить любую языковую модель (LLM) в контекст кодовой базы для дальнейшей разработки, рефакторинга или поиска ошибок.

## 1. Общая архитектура проекта

Бот написан на **Python** и предназначен для сеточной торговли (Grid Trading) деривативами (USDT Perpetuals) на бирже **Bybit** через официальную библиотеку `pybit` (V5 API). 

Проект состоит из следующих основных компонентов (файлов):
- **`main_grid.py`**: Главный цикл программы (Event Loop). Управляет инициализацией, проверкой условий запуска (RSI), ребалансировкой, мониторингом просадки (Drawdown) и остановкой.
- **`grid_engine.py`**: Математическое ядро сетки. Отвечает за расчет уровней (calculate_levels), расчет объемов ордеров, проверку пересечений цены (check_execution), сохранение/загрузку состояния в JSON и логику смещения сетки (recenter_grid).
- **`grid_executor.py`**: Класс для взаимодействия с API Bybit. Управляет выставлением, отменой ордеров, получением баланса, цены, свечей (klines) и расчетом индикаторов (ATR, RSI). Отправляет уведомления в Telegram.
- **`grid_config.py`**: Файл конфигурации, загружающий переменные из `.env`. Содержит все торговые настройки (символ, плечо, диапазоны, флаги умных функций).
- **`dashboard.py`**: Легковесный веб-интерфейс на базе **Streamlit** для мониторинга активных ордеров, общей прибыли и состояния сетки локально.
- **`state.json`**: Файл, в котором `grid_engine` хранит свое состояние (уровни, прибыль, количество сделок) для восстановления после перезапуска.

## 2. Ключевые Фичи (Smart Features)

### 2.1 Динамический Шаг Сетки (ATR)
- Если в конфиге включен `USE_ATR_STEP`, бот перед стартом (или при ребалансировке) запрашивает исторические 15-минутные свечи, рассчитывает индикатор **ATR (Average True Range)** и умножает его на `ATR_MULTIPLIER`.
- Полученное значение становится новым `GRID_RANGE_PCT` (в процентах). Это позволяет растягивать сетку при высокой волатильности (чтобы избежать быстрого выхода цены за границы) и сужать при низкой (чтобы чаще собирать профит).

### 2.2 Умный Вход (RSI Smart Entry)
- Защита от запуска покупок на самых "хаях" или шортов на "лоях".
- Если `RSI_ENTRY_CHECK = True`, бот в `main_grid.py` уходит в цикл ожидания (while).
- Он запрашивает 15-минутный RSI.` Если RSI находится в нейтральной зоне (от 30 до 70), бот запускает сетку. Если актив перекуплен (>70) или перепродан (<30), бот ждет.

### 2.3 Авто-ребалансировка (Recenter Grid)
- Если цена выходит за пределы верхней или нижней границы сетки (`GRID_UPPER`, `GRID_LOWER`), бот вызывает `_do_rebalance`.
- Параметр `REBALANCE_MODE` определяет поведение со старыми позициями:
  - `CLOSE_ALL`: Жестко закрывает все текущие позиции по рынку, фиксируя плавающий убыток или прибыль, и строит новую сетку вокруг новой цены.
  - `HOLD_POSITION`: Оставляет открытые позиции (надеясь на возврат), но перестраивает лимитные ордера вокруг текущей цены.

### 2.4 Auto-Compound (Реинвестирование)
- Если `AUTO_COMPOUND = True`, бот при каждом обновлении / ребалансировке берет реализованную прибыль из `state.json` (`total_profit`) и прибавляет её к базовому `INVESTMENT`.
- Размеры новых лимитных покупок/продаж геометрически возрастают по мере успешной работы сетки. 

### 2.5 Защита Капитала (Risk Management)
- **Max Drawdown**: Бот постоянно оценивает плавающий PnL. Если убыток достигает `MAX_DRAWDOWN_PCT`, вызывается `_emergency_stop()` (закрытие позиций, отмена ордеров, выход из программы).
- **Funding Rate Protection**: Бот проверяет ставку финансирования (Фандинг) на деривативах. Если абсолютное значение ставки превышает `MAX_FUNDING_RATE`, бот экстренно останавливается, чтобы избежать "сжигания" баланса на комиссиях за удержание позиций.

## 3. Механизм работы Основного Цикла (main_grid.py)

1. **Инициализация**: Сброс старой сетки, загрузка состояния или первоначальная настройка. Кросс-маржа, плечо.
2. **Проверка входа**: Цикл RSI.
3. **Расчет уровней**: `engine.calculate_levels` и выставление лимитных ордеров (через `executor.place_order`).
4. **Главный цикл `while self.running`**:
    - Опрашивает `get_price()`.
    - Проверяет `engine.check_execution()`. В случае исполнения ордера рассчитывает профит, создает парный ордер с другой стороны.
    - Проверяет границы (если цена < нижнего лимита или > верхнего). Если да -> вызывает `_do_rebalance()`.
    - Проверяет `Funding Rate` и `Drawdown`.
    - Спит (time.sleep).

## 4. Внедренные улучшения и Статус разработки
- **База данных (Успешно внедрено)**: Сохранение состояния переведено с уязвимого конфигурационного файла `state.json` на транзакционную базу данных SQLite (`grid_state.db`). Дашборд на `streamlit` (`dashboard.py`) также читает данные напрямую из таблицы БД в реальном времени.
- **Логирование (Успешно внедрено)**: Вызовы принтов `print()` удалены из ядра. Развернута система структурированного логирования через модуль внутри `logger.py`, которая фиксирует все события как в файлы (`logs/...`), так и в консоль с поддержкой ротации.
- **Тестирование (Demo Mode / Обход API)**: Для безопасного локального запуска без реальных ключей API Bybit внедрен функционал обхода. При запуске через `python main_grid.py --demo` используется публичный клиент для запросов рыночных цен через `get_tickers`, а приватные вызовы (сделки и аккаунт) используют тестовую заглушку внутри `grid_executor.py`. Это позволяет тестировать `GridEngine` и `dashboard.py` без ошибки инициализации сессии.

## 5. Что стоит иметь в виду ИИ при дальнейшей доработке
- **Синхронность**: Сейчас бот всё ещё использует `time.sleep()` и синхронные HTTP-запросы библиотеки `pybit`. Для одного символа (пары) это работает отлично, но для превращения бота в мульти-символьного (торговля 10+ монетами) потребуется переписать сеть на модули асинхронного программирования `asyncio` вместе с асинхронным клиентом `pybit`.
